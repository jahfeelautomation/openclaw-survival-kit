# v0.6 — Community Triage Task (design doc, pre-implementation)

Status: **design, not yet built.** Implementation is intentionally deferred until we have at least one real incoming report to validate the dedup logic against. Writing the dedup rules against guessed reports will produce rules that don't match how actual reports look.

Owner: Claude (in Cowork). Reviewed by: JahFeel, Jeff.

## Why this exists

v0.5 gave users a 30-second way to file a structured report when a claw-medic fix didn't work. v0.6 closes the loop: the existing 5am daily task (`openclaw-survival-kit-daily`) gets extended to process those reports so the kit gets smarter from real-world data instead of our internal guesses.

The existing 5am task already reads Jeff's memory + EVENTS.jsonl, files GitHub issues for new bugs on the kit repo, drafts PRs (never auto-pushes for the first 30 days), and drafts a daily tweet for JahFeel approval. v0.6 adds a new phase at the start of that same task.

## What v0.6 adds to the 5am task

A new **INBOX** phase that runs first, before any of the existing phases.

```
INBOX   → fetch newly-filed issues on the kit repo, categorize each one,
          draft a response, append to the daily report
MEMORY  → (existing) scan Jeff's memory / events for new bugs
ISSUES  → (existing) file GitHub issues for new bugs we found
PRs     → (existing) draft PRs for fixes we have solutions for
TWEET   → (existing) draft daily tweet for JahFeel
```

### INBOX phase in detail

**1. Fetch new reports.**
The task holds a small state file at `~/.openclaw/kit-triage-state.json` with `last_seen_issue_number` and `last_run_utc`. Each run, it uses `gh api repos/jahfeelautomation/openclaw-survival-kit/issues` with `since=<last_run_utc>` and filters to issues with label `fix-failure` OR `new-bug` that were created OR updated since last run, and whose number is higher than last_seen_issue_number (new) or is already in the triage tracking table (updated).

**2. For each issue, build a dedup key.**
The key is deterministic and comparable across reports. Revised in review (Jeff, 04/14): outcome text is dropped from the machine dedup entirely because user phrasing varies too much to hash reliably. The key groups reports into CANDIDATE buckets; the triage maintainer (or a later heuristic) decides whether distinct outcomes within a bucket are actually the same failure or not.

```
dedup_key = sha256(
    normalize(fix_name) +
    "|" + claw_medic_version +
    "|" + os_family             # "windows" / "macos" / "linux", not the patch level
).hexdigest()

def normalize(s: str) -> str:
    # Exact, locked pseudocode — implementations must match this byte-for-byte.
    if s is None:
        return ""
    s = s.strip().lower()
    s = s.replace('"', '').replace("'", "")
    s = " ".join(s.split())       # collapse all whitespace runs to single space
    return s
```

Outcome text still lands in the daily triage report for human review, and the report groups all reports sharing a dedup_key into one section. That way the maintainer sees "3 reports against fix `gateway_process` on Windows / claw-medic 0.5" with the three outcome strings underneath, and decides whether to treat them as one issue or three. Far more reliable than trying to bucket outcome phrasing with a 12-word prefix.

**3. Route by category.**

| Category | Triggers | Draft action |
|---|---|---|
| `duplicate` | `dedup_key` matches an issue we've already processed | Draft a reply comment linking to the original + the status. **Closes the new issue, no merge activity.** |
| `stale-version` | `claw_medic_version < latest_released_version` (no "did this fix_name change" condition — we don't track version-to-fix-name mapping and Jeff correctly flagged that overclaiming is worse than underclaiming) | Draft a reply: "You're on claw-medic `vX.Y`; latest is `vN.N` — [release notes](https://github.com/jahfeelautomation/openclaw-survival-kit/releases/tag/vN.N). Upgrade (`git pull`), re-run `claw-medic --report` if still broken, and we'll take a fresh look." **Does not close.** |
| `new-signal` | First time we've seen this dedup key AND version is current | Draft a PR: add a new check OR tighten an existing check's condition. Comment on the issue with a link to the draft PR. **PR is draft-only; maintainer merges manually.** |
| `needs-more-info` | Required fields empty (fix-name or outcome) OR JSON attachment missing | Draft a reply asking for the specific missing field. Add label `needs-info`. **Does not close.** |
| `spam` | ALL of: author has zero commits to this repo, AND (account <30 days old OR body contains 3+ URLs with no readable sentence structure), AND title/body matches a known-spam pattern (crypto, "buy followers", link farms). Two of the three gates must trip, not just one. | Draft a label-as-spam + close action. **Flag for maintainer to review before executing** — spam heuristics are historically noisy and we'd rather let a borderline case through than autoclose a frustrated real user. |

**4. Output.**
Everything the INBOX phase does lands in `reports/YYYY-MM-DD-triage.md` with one section per issue. The maintainer (JahFeel, via Jeff on cron) opens the report, reviews each draft, and either approves (Jeff `gh` pushes the comment / PR) or overrides.

**Nothing is auto-posted.** Every action is a draft until approved. Same rule as the existing PR/tweet phases.

### State tracking

`~/.openclaw/kit-triage-state.json`:

```json
{
  "last_run_utc": "2026-04-14T17:30:00Z",
  "last_seen_issue_number": 7,
  "processed": [
    {
      "issue_number": 3,
      "dedup_key": "abc123...",
      "category": "new-signal",
      "action_drafted": "PR #12",
      "outcome": "merged",
      "processed_at_utc": "2026-04-10T05:12:00Z"
    },
    {
      "issue_number": 5,
      "dedup_key": "abc123...",
      "category": "duplicate",
      "action_drafted": "reply-then-close",
      "linked_to_issue": 3,
      "processed_at_utc": "2026-04-12T05:12:00Z"
    }
  ]
}
```

State is append-only. We never retroactively re-categorize. If the maintainer overrides a draft (e.g. marks our "duplicate" as actually a new signal), we capture that as a correction in the next run's report but don't mutate the original entry. That's how the triage task learns.

### What v0.6 does NOT do

- **No auto-merge.** Every PR is draft; the maintainer merges.
- **No auto-close.** Duplicate-category items are drafted to close; maintainer applies.
- **No re-opening closed issues.** If a user re-opens a closed issue, we treat it as a fresh signal.
- **No cross-repo PRs.** The draft PRs are against the kit repo only, not against upstream OpenClaw. That's a different loop we're not wiring here.
- **No rate-limiting bypass.** If we hit GitHub API rate limits, the task retries ONCE after 60 seconds (Jeff's suggestion — cheap, handles transient secondary-rate-limit bumps), then fails loudly and waits for the next scheduled run. No back-off loop.
- **Auth check first.** The INBOX phase starts with `gh auth status` — if token is expired or missing, the task exits with a clear error before touching anything else. Fail-fast prevents half-processed runs (Jeff).

## Calibration period (why we're waiting on real data)

The dedup logic looks reasonable on paper but has a lot of knobs: `first_N_words` N, which fields get normalized, what counts as "same OS family". Writing those knobs before seeing one real report will produce values that work on imagined reports and fail on actual ones.

Concrete trigger to build:
- 1 real fix-failure report from a non-Anthropic/non-JahFeel account hits the repo, OR
- 7 days pass with zero reports (in which case the trigger for building shifts — we'd need to investigate why no one is using it before extending the triage system).

During the wait, JahFeel and Jeff should treat any real report that arrives as a manual dry-run of the triage logic: read the issue, try to categorize it using the table above, note what would need to change in the dedup key. Those notes go at the bottom of this doc under "Calibration notes" and directly inform the implementation.

## Implementation path (when we go)

1. Write a standalone Python module `triage/` in the kit repo: `fetch_issues.py`, `dedup.py`, `categorize.py`, `draft_responses.py`. No runtime deps beyond `gh` CLI + stdlib. Unit-tested in isolation.
2. Extend the existing `openclaw-survival-kit-daily` scheduled task prompt to run INBOX phase first, producing `reports/YYYY-MM-DD-triage.md` before the existing phases.
3. Dry-run on any backlog that exists at that point — generate the report but don't act. Have JahFeel/Jeff review.
4. Flip "draft responses" to "draft + execute-on-approval" only after 2 weeks of review-and-agree with 0 maintainer overrides on spam / duplicate categories.

## Open questions (answered in review, 04/14)

- ~~Should `stale-version` also link to the specific commit that changed the behavior?~~ **Answered (Jeff):** Link to the release tag instead — `https://github.com/jahfeelautomation/openclaw-survival-kit/releases/tag/v0.X`. Version-to-commit indexing is more infrastructure than it's worth; release tags are close enough and we already create them.
- Do we want anonymized aggregate stats in the daily tweet? ("This week: 4 fix-failure reports, 3 resolved by v0.N") — nice PR, but premature without consent language in the report template. Revisit after 30 days.
- ~~At what point does an issue stop being "new-signal" and become its own mini-project?~~ **Answered (Jeff):** When the same `dedup_key` hits 3+ times AND we've shipped 2+ claw-medic versions without an addressable fix. At that point, promote to a pinned issue + a workaround doc in `docs/workarounds/`. Stop drafting per-report PRs against it; instead, new reports get auto-linked to the pinned issue as duplicates.
